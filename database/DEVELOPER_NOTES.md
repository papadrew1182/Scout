# Scout Life Management: Developer Notes

How task generation, daily win scoring, and weekly payout computation work.

## 1. Task Generation

Task instances are generated by a scheduled process (e.g., daily at midnight in the family's timezone, or on-demand when a user opens the app for the day).

### Routine-sourced task_instances

For each active routine where the current day matches the recurrence pattern:

1. Look up the routine's `family_member_id`, `block`, and applicable due time (`due_time_weekday` or `due_time_weekend`).
2. Combine `instance_date` + `due_time` + family timezone to produce `due_at` as `timestamptz`.
3. Insert one `task_instances` row with `routine_id` set and `chore_template_id` NULL.
4. For each active `routine_steps` row belonging to that routine, insert one `task_instance_step_completions` row with `is_completed = false`.

### Chore-sourced task_instances

For each active chore_template where the current day matches the recurrence pattern:

1. Evaluate `assignment_type` and `assignment_rule` to determine which family_member(s) get a task_instance today:
   - `fixed`: always the same member.
   - `rotating_daily`: check day-of-month parity. Odd day = one member, even day = another. Read from `assignment_rule.odd` / `assignment_rule.even`.
   - `rotating_weekly`: check the rotation schedule (e.g., 8-week swap cycle). Read `current_owner` and `rotation_started` from `assignment_rule`.
2. Combine `instance_date` + `due_time` + family timezone to produce `due_at`.
3. Insert one `task_instances` row per assigned member with `chore_template_id` set and `routine_id` NULL.

### Dog walk assistant pattern

The "Dog Walks (Sadie lead)" chore_template generates a task_instance for Sadie every day. The "Dog Walk Assistant (Willie)" chore_template generates a separate task_instance for the rotating assistant (Townes or River by day parity). Both are independent task_instances with their own completion tracking and Daily Win contribution.

### Idempotency

Before inserting, check the unique constraints:
- `(family_member_id, routine_id, instance_date)` for routine instances
- `(family_member_id, chore_template_id, instance_date)` for chore instances

If a row already exists for today, skip it. Use `INSERT ... ON CONFLICT DO NOTHING` or a pre-check query.

## 2. Daily Win Computation

A Daily Win is computed after all deadlines for a given day have passed (e.g., at 10:00 PM or on the next morning's generation run).

### Algorithm

For a given `family_member_id` and `instance_date`:

1. Query all `task_instances` for that member and date.
2. For each instance, determine effective completion:
   - If `override_completed IS NOT NULL`, use `override_completed` as the completion status.
   - Otherwise, use `is_completed`.
3. Additionally, check deadline compliance: `completed_at <= due_at` (or override is set).
4. If ALL task_instances for that member on that date are effectively completed by deadline: `is_win = true`.
5. If ANY task_instance is not completed or was completed after deadline (without override): `is_win = false`.
6. Record `task_count` (total instances) and `completed_count` (effectively completed instances).
7. Insert or update the `daily_wins` row.

### Weekend exclusion

Only compute and insert `daily_wins` rows for Monday through Friday. The database enforces this via `CHECK (EXTRACT(isodow FROM win_date) BETWEEN 1 AND 5)`. Saturday Power 60 and other weekend task_instances exist for tracking but do not affect scoring.

## 3. Weekly Payout Computation

Weekly payouts are computed after all five daily_wins for the week are materialized (e.g., Friday night or Saturday morning).

### Algorithm

For a given `family_member_id` and `week_start` (Monday date):

1. Count daily_wins where `is_win = true` for that member in the date range `[week_start, week_start + 4 days]`.
2. Apply the payout tier:
   - 5 wins = 100% of baseline
   - 4 wins = 80%
   - 3 wins = 60%
   - 2 or fewer = 0%
3. Look up the member's baseline weekly allowance (application config, not in the database yet — Sadie $12.00, Townes $9.00, River $7.00).
4. Compute `amount_cents = baseline_cents * tier_percentage`.
5. Insert one `allowance_ledger` row with `entry_type = 'weekly_payout'` and `week_start` set.

### Duplicate prevention

The partial unique index `(family_member_id, week_start) WHERE entry_type = 'weekly_payout'` prevents double payouts. Use `INSERT ... ON CONFLICT DO NOTHING` or check before inserting.

### School rewards

Entered manually by a parent. Insert one `allowance_ledger` row with `entry_type = 'school_reward'`, `amount_cents` set to the reward value, `week_start` NULL (not tied to a scoring week), and a descriptive `note`.

### Balance

A member's current balance is `SELECT SUM(amount_cents) FROM allowance_ledger WHERE family_member_id = $1`. No separate balance column exists — it is always derived.
