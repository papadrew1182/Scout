## 3. Critical path

**Phase 1 - Core engine.** Lands the parent-child dispatch model, three built-in scanners (`overdue_task`, `upcoming_event`, `missed_routine`), scheduler job, Inbox row, push delivery, and source-level dedupe. Everything downstream hangs off `nudge_dispatches` plus `nudge_dispatch_items`.

**Phase 2 - Quiet hours + batching.** Adds the gating layer. `quiet_hours_family` table, optional per-member override in `member_config['nudges.quiet_hours']`, and a 10 minute batch window that collapses multiple source items into one Inbox row + one push. Must ship before P3 so the composer sees the final batched summary, not the pre-batch individual proposals.

**Phase 3 - Personalized copy.** Swaps fixed templates for AI-composed body via orchestrator, using the member's personality preamble. Fallback to P1 fixed templates when AI is unavailable, over cap, moderation-rejected, or times out under 3 seconds. Dedicated orchestrator entry point, not shared with ScoutSheet.

**Phase 4 - Admin rule engine.** Opens `scout.nudge_rules` CRUD at `/api/admin/nudges/rules`. The blocker is not CRUD. It is safe SQL validation and safe execution. Rules flow through the same `apply_proactivity` -> quiet-hours -> batching -> dispatch pipeline.

**Phase 5 - AI-driven discovery.** Hourly tick calls `nudge_ai_discovery.propose_nudges`. The family-state digest is built from approved structured domain tables only. It rides the existing dispatch pipeline so quiet hours, batching, personalization, and dedupe all apply to AI-suggested nudges automatically.

## 4. Core data model and gate order

### Core data model

- `scout.nudge_dispatches` = one delivered artifact. One row corresponds to one Inbox entry and at most one push.
- `scout.nudge_dispatch_items` = one source proposal attached to a dispatch. This is where source provenance and dedupe live.
- `deliver_after_utc` controls when a dispatch may actually be sent.
- `occurrence_at_utc` and `occurrence_local_date` capture the original source event and never change when quiet hours delay delivery.
- `source_dedupe_key` format: `{member_id}:{trigger_kind}:{trigger_entity_id}:{occurrence_local_date}`.
- Parent dispatch rows are only written if at least one child source item inserts successfully.

### Applied in this order on every proposal

1. Member `proactivity` setting.
   - `quiet` drops the proposal outright.
2. `forthcoming` doubles lead time and can fire earlier.
   - `balanced` uses default lead time.
3. Resolve immutable source fields.
   - compute `occurrence_at_utc`
   - compute `occurrence_local_date`
   - compute `source_dedupe_key`
4. Optional best-effort dedupe prefilter.
   - may check for existing `source_dedupe_key` to save work
   - never treated as the correctness boundary
5. Quiet-hours gate (P2 onward).
   - per-member `member_config['nudges.quiet_hours']` override wins over family default
   - inside window: `low` severity is dropped with `suppressed_reason='quiet_hours'`
   - inside window: `normal` severity is held until the window end by moving `deliver_after_utc`
   - inside window: `high` severity delivers anyway
6. Batch window grouping (P2 onward).
   - proposals for the same member inside 10 minutes and sharing the same effective delivery window collapse into one composite dispatch
7. Body composition.
   - P1-P2: fixed template
   - P3 onward: AI-composed with fallback to fixed template on any failure condition
8. Transactional write.
   - create parent `nudge_dispatches` row candidate
   - insert one or more `nudge_dispatch_items` rows
   - `UNIQUE (source_dedupe_key)` on child rows is the authoritative dedupe boundary
   - if all child rows conflict, abort the parent dispatch write
9. Write `parent_action_items` row and call `push_service.send_push` if the member has an active device.

## 5. Risks being tracked

**Tick time under load.** P1 and P4 both add DB work to the 5 minute tick. P1 acceptance must measure under a realistic fixture (>=50 `personal_tasks`, >=20 upcoming events, active family) and assert total tick time <10 seconds. P4 must add hard kill switches, not just benchmarking:
- per-rule `statement_timeout`
- per-rule `lock_timeout`
- total rule-scan budget for the tick
- stop scanning remaining rules once budget is exhausted

**SQL validation and execution (P4).** This remains the hardest thing in the sprint. V1 safety bar:
- Parse with `sqlglot.parse(template_sql, read="postgres")`.
- Reject unless there is exactly one statement and the root node is `SELECT`.
- V1 allowed shape: `SELECT ... FROM ...` with optional `JOIN`, `WHERE`, `GROUP BY`, `HAVING`, `ORDER BY`, `LIMIT`.
- Reject `WITH`, `UNION`, `INTERSECT`, `EXCEPT`, subqueries, `COPY`, multiple statements, comments, DML, DDL, `CALL`, and any node type outside the allowed subset.
- Reject any identifier in `pg_*`, `information_schema`, or any unapproved schema/table/view.
- Maintain a small function allowlist only (proposal: `now`, `coalesce`, `lower`, basic date/time ops, `count`, `min`, `max`).
- Re-serialize canonical SQL from the validated AST and execute only that canonical SQL, never the raw `template_sql` string.
- Execute under a least-privilege DB role in a `READ ONLY` transaction.
- Set `statement_timeout` and `lock_timeout` locally for every rule query.
- Attack suite must land before the CRUD routes are wired.

**Cross-midnight dedupe drift.** Quiet hours can push a normal-severity dispatch past midnight. That is now tracked explicitly by separating `occurrence_local_date` from `deliver_after_utc`. Smoke coverage must prove that a held nudge delivered the next morning does not create a second source item for the same original occurrence.

**Dedupe collision across P1 and P5.** P1 built-ins and P5 AI discovery now collide at `nudge_dispatch_items.source_dedupe_key`. That is the intended behavior and must be asserted explicitly in P5 smoke coverage.

**AI fallback matrix (P3).** Four fallback conditions remain mandatory test cases: `ai_available` false, weekly soft-cap hit, moderation rejection, timeout <3 seconds.

**Personalized copy drift (P3).** Keep the substring matrix across at least 3 personality preambles (`terse`, `warm`, `formal`) and assert that tone markers still hold.

**Sensitive-domain visibility (P5).** `nudges.view_own` is broadly granted, including `YOUNG_CHILD`. No health or finance driven nudges are allowed into P5 by default until capability gates and a per-recipient visibility matrix exist.

## 6. Amendments the Claude Code instructions will carry

- Dedupe resolution is now based on immutable `occurrence_local_date`, not `scheduled_for::date`.
- Batching uses a parent-child model: one `nudge_dispatches` row, one or more `nudge_dispatch_items` rows.
- `source_dedupe_key` uniqueness on `nudge_dispatch_items` is the source of truth for dedupe. Any pre-check is an optimization only.
- Quiet hours may delay delivery by changing `deliver_after_utc`, but they never mutate `occurrence_at_utc` or `occurrence_local_date`.
- `quiet_hours_family` stays as a dedicated table by design. Document it as an intentional exception to the `family_config` convention.
- Hearth Display is not a nudge surface. Nudges are Inbox + push only. Google Calendar sync behavior is unchanged by this sprint.
- AI discovery state digest is structured-domain-tables only. It explicitly excludes `ai_messages`, YNAB, and Apple Health in this revision.
- Admin UI route namespace is `/admin/ai/*`. Quiet hours and rules share `/admin/ai/nudges`.
- `sql_template` path ships in P4 only with Postgres-dialect AST validation, canonical SQL execution, least-privilege DB role, and strict query timeouts.
- Rules run after built-ins on the shared 5 minute tick only while they remain within the configured per-rule and total-job budgets.
- `No em dashes` applies to AI-composed copy. Add that constraint to the P3 composer system prompt.
- Severity assignment for built-in scanners remains the current proposal unless Andrew changes it:
  - `overdue_task` -> `normal`
  - `upcoming_event` -> `normal`
  - `missed_routine` -> `low`
