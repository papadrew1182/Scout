## Blocker resolutions in this revision

**Blocker 1 - scout_scheduled_runs schema mismatch (both scripts).**

Original code queried `run_started_at` and `run_ended_at` columns. Those don't exist. Schema verified against migration 016 (`backend/migrations/016_tier1_proactive.sql:22-36`) and ORM at `backend/app/models/scheduled.py:13-36`. No later migration modifies this table (grep confirmed). Actual columns:

```
id uuid, job_name text, family_id uuid, member_id uuid nullable,
run_date date, status text CHECK IN ('success','error','skipped'),
duration_ms integer, result jsonb, error text, created_at timestamptz
```

Write pattern (`backend/app/scheduler.py:311-358`): jobs INSERT the mutex row with status='success' upfront via a savepointed INSERT, do their work, then UPDATE duration_ms/result/error/status in place. The outer tick commits after the job function returns. Observers see rows only after commit, which is after work finished. There is no mid-flight state in this table.

Corrections:
- `quiesce_prod.py::poll_scheduler_idle` - dropped the `run_started_at IS NOT NULL AND run_ended_at IS NULL` query entirely. Now checks only "zero rows with `created_at > baseline` for the full 5-minute window." On failure, surfaces `job_name`, `status`, `created_at`, `error` for each offending row so the operator can investigate fast.
- `unquiesce_prod.py::poll_for_healthy_tick` - rewrote around the status column. Rules:
  - status='success' with `created_at > baseline` = healthy tick, pass.
  - status='error' with `created_at > baseline` = failed tick, fail verification, surface the `error` column.
  - status='skipped' = does NOT count, keep polling until success or deadline.

Code docstrings in both scripts now cite the authoritative schema source.

## What shipped in this PR

### Scripts

**`scripts/quiesce_prod.py`** - verify-only. Flow:

1. Smoke-test `/ready` and `SCOUT_DATABASE_URL` connectivity.
2. Print current `/ready` snapshot for operator reference.
3. Prompt operator to set 3 env vars on Railway: `SCOUT_SCHEDULER_ENABLED=false`, `SCOUT_ENABLE_BOOTSTRAP=true`, `SCOUT_ENABLE_AI=false`. Wait for Enter.
4. Verify via `/ready`: `bootstrap_enabled=true`, `ai_available=false`.
5. Re-prompt operator to confirm `SCOUT_SCHEDULER_ENABLED=false` specifically (guards against a slow dashboard flip racing the idle poll). Wait for Enter.
6. 5-minute poll of `public.scout_scheduled_runs`. Fails if any new rows appear after the baseline timestamp, OR any in-flight rows exist (`run_started_at IS NOT NULL AND run_ended_at IS NULL`).
7. Purge Supabase Storage attachments bucket: POST `/storage/v1/object/list/{bucket}` to enumerate, then DELETE `/storage/v1/object/{bucket}` with prefixes list. Re-lists after delete to confirm bucket is empty.

Idempotent. Running twice re-prompts, re-verifies, and re-purges an already-empty bucket without error.

**`scripts/unquiesce_prod.py --ai-only | --full`** - verify-only, exactly one flag.

- `--ai-only`: prompt operator to set `SCOUT_ENABLE_AI=true`, wait for Enter, poll `/ready.ai_available` until `true` (90s timeout). Touches nothing else.
- `--full`: prompt operator to set `SCOUT_SCHEDULER_ENABLED=true` AND `SCOUT_ENABLE_BOOTSTRAP=false`, wait for Enter, poll `/ready.bootstrap_enabled` until `false`, then poll `public.scout_scheduled_runs` for up to 6 minutes for at least one healthy tick (new row with both `run_started_at` and `run_ended_at` set). Fails if no healthy tick lands within the window.

Rejects calls with no flag. Argparse's mutually_exclusive_group rejects calls with both flags.

### Env var name correction

The v5.1 plan and the re-enable checklist I shipped in PR 0.1 used `SCOUT_AI_ENABLED`. The actual env var is `SCOUT_ENABLE_AI` (feeds `settings.enable_ai` at `backend/app/config.py:93`; env_prefix is `SCOUT_`). Setting the wrong name has no effect - AI stays on, belt-and-suspenders silently fails.
