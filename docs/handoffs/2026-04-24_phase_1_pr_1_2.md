## What shipped in this PR

### Migration (mirrored to `backend/migrations/` and `database/migrations/`)

**`056_phase1_truncate_domain_data.sql`** - TRUNCATEs 55 targets (53 retained scout.* tables + `public.sessions` + `public.scout_scheduled_runs`) and verifies the outcome in three inline checks. Transactional; any RAISE EXCEPTION rolls back the whole migration.

**Structure:**

1. **Single `TRUNCATE TABLE ... CASCADE` statement** enumerating all 55 targets. CASCADE is a safety net; all known inter-scout FKs are within the enumerated set.
2. **Set-match check** (`DO` block): queries `information_schema.tables` for `table_schema = 'scout' AND table_type = 'BASE TABLE'` and asserts the result equals the 53-table enumeration. Any difference - unexpected table present, or enumerated table missing - raises an exception. Added at Andrew's request during scope review to guard against CASCADE silently truncating an unlisted table.
3. **Zero-row verification** (`DO` block): loops over all 55 truncated targets, runs `SELECT COUNT(*)`, raises on any non-zero result. Honors plan's "SELECT COUNT(*) per truncated table within same transaction. No `pg_stat_user_tables`."

## Consumer audit

### DB-side dependencies

**Views:** none. PR 1.1 dropped the 10 views that depended on truncated tables (`v_household_today`, `v_rewards_current_week`, 8 shim views). The 2 kept views (`v_calendar_publication`, `v_control_plane`) will return 0 rows after TRUNCATE since their underlying tables are empty - this is expected behavior, not a structural issue.

**Triggers:** kept triggers on truncated tables fire during the truncate only if per-row (`FOR EACH ROW`), which none are (all are `FOR EACH STATEMENT`, e.g. `set_updated_at()` calls - irrelevant for TRUNCATE). No cascaded trigger surprises.

**Sequences / identity columns:** none (all PKs use `uuid DEFAULT gen_random_uuid()` per pre-flight Part 5.2). TRUNCATE ... RESTART IDENTITY is not needed and is not used.

**Stored procedures / functions:** none invoke the truncated tables at schema level. The shared `set_updated_at()` function is preserved and unaffected.

**FKs:** PR 1.1 migration 054 dropped 65 cross-schema FKs from scout.* into the PR 1.3 drop set. All remaining FKs on the 53 retained tables point at other retained tables within the same set, which are all included in the single TRUNCATE ... CASCADE. Two kept public.* tables have no incoming FKs (verified via grep on pg_dump snapshot).

## Open items flagged

**These are planning tasks for subsequent PRs, not PR 1.2 blockers.**

### O1. PR 1.3 must drop the public.sessions -> public.user_accounts FK before dropping public.user_accounts

`public.sessions` has outgoing FK `sessions_user_account_id_fkey FOREIGN KEY (user_account_id) REFERENCES public.user_accounts(id) ON DELETE CASCADE` (verified in `docs/plans/_snapshots/2026-04-22_pre_rewrite_full.sql`). PR 1.3 drops `public.user_accounts`. Without dropping this FK first, the table drop fails. Plan forbids CASCADE.

Pre-flight Part 1 did not surface this (same category as the 20-table FK gap found during PR 1.1 execution). `public.sessions` is kept per plan §3 with "Schema preserved", but the FK into `public.user_accounts` is part of that schema and will become invalid when `user_accounts` drops.

**Action for PR 1.3 planning:**
- Add `ALTER TABLE public.sessions DROP CONSTRAINT IF EXISTS sessions_user_account_id_fkey;` to PR 1.3's FK-drop stage.
- Decision needed: does the FK get recreated in Phase 2 PR 2.1 (identity rebuild) to point at `scout.user_accounts`, or does `public.sessions` no longer hold the FK long-term?
- Plan §3 says schema is preserved; a literal reading means the FK stays but points at the rebuilt table. Phase 2 PR 2.1 would need to add `ALTER TABLE public.sessions ADD CONSTRAINT sessions_user_account_id_fkey FOREIGN KEY (user_account_id) REFERENCES scout.user_accounts(id) ON DELETE CASCADE;`. Needs Andrew's call.

### O2. Re-scan for more Part-1-gap FKs into the full PR 1.3 drop set

PR 1.1 surfaced 20 scout.* tables with FKs into the PR 1.3 drop set that pre-flight Part 1 missed. PR 1.2 now surfaces another (`public.sessions`). A more conservative exercise before PR 1.3 opens: scan the pg_dump snapshot for every cross-FK into `public.user_accounts`, `public.role_tiers`, `public.meals`, `public.parent_action_items`, and any other PR-1.3-drop target, from ANY source (not just scout.*), to catch anything else the pre-flight missed.
