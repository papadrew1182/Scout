## What shipped in this PR

### Migration (mirrored to `backend/migrations/` and `database/migrations/`)

**`057_phase1_drop_public_legacy.sql`** - drops 39 public.* legacy tables in topological order. Single transaction with three steps:

1. **Step A: 4 explicit FK drops.** 3 on kept public.* tables (sessions and scout_scheduled_runs) for the cross-FK problem the pre-flight Part 1 didn't surface; 1 mutual FK breaker (`purchase_requests_linked_grocery_item_id_fkey`) for the grocery_items <-> purchase_requests circular reference. Rule for choosing which side of a mutual pair to drop within an alphabetically-ordered tier: drop the FK on the alphabetically-LATER table. That way the earlier table can drop cleanly first (taking its still-present FK with it), then the later table drops with no incoming refs.
2. **Step B: 39 DROP TABLE statements** in 5-tier topological order (no CASCADE).
3. **Step C: Set-match `DO` block** asserting the post-drop public.* BASE TABLE set equals exactly {_scout_migrations, sessions, scout_scheduled_runs}. Same pattern as 056's scout-side set-match check.

Any RAISE EXCEPTION rolls back the whole migration. Idempotent (`IF EXISTS` everywhere).

### Plan supersession note

**Plan Phase 1 PR 1.3 step list incorrect against actual FK graph; superseded by 5-tier topological order in this migration.**

The plan's stated 8-step order had two ordering bugs surfaced by the O2 full-snapshot scan:

- **Bug 1:** `public.events.events_task_instance_id_fkey -> public.task_instances` (ON DELETE SET NULL). PG blocks `DROP task_instances` while events's FK exists, regardless of ON DELETE. Plan put task_instances at step 2 and events in step 8 ("any order"); the step 2 drop would fail.
- **Bug 2:** `public.ai_tool_audit`, `ai_homework_sessions`, `family_memories`, `planner_bundle_applies` all have SET NULL FKs to `public.ai_conversations`. Plan put ai_conversations at step 7 and the four referencers in step 8; the step 7 drop would fail. Plan only flagged `ai_messages` (CASCADE) as needing to precede ai_conversations.

The 5-tier topological order in 057's Step B is the minimum rearrangement that satisfies the actual FK graph derived from the pg_dump snapshot. No upstream plan-text patch proposed (per Andrew's direction during scope review).

## No-CASCADE compliance statement

This migration uses `DROP TABLE IF EXISTS` (not `DROP TABLE ... CASCADE`) for every table drop. Every blocking FK is handled either:

1. By an explicit `ALTER TABLE ... DROP CONSTRAINT IF EXISTS` in Step A (4 FKs total: 3 on kept public.* tables, 1 mutual breaker), or
2. By topological tier ordering in Step B (each FK source drops before its target, taking the FK with it).

Any DROP TABLE failure means an uncaught dependency exists; the transaction rolls back; we hit a v5.1 section 8 stop-condition.
