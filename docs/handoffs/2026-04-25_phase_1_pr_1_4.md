### Why edit 057 in place rather than ship a new 058

PR 1.3 review (post-bug-detection) explicitly compared two options:

- **A1: Edit 057 in place.** Atomic rollback meant 057 has no row in `_scout_migrations` and Step A's FK drops were also reverted; the database is in a "057 never happened" state. Editing the migration file before any environment durably applies it is completing the work, not rewriting history.
- **A2: Add a new 058 with the corrected sequence.** Leaves broken 057 in the migration file tree, which is a landmine for any fresh-DB rebuild (CI, dev onboarding, disaster recovery): `migrate.py` runs migrations in numerical order, so a fresh DB hits broken 057 before reaching corrective 058 and fails. A2 only "works" on the prod DB which already executed migrations 001-056; it would silently break every other environment.

Andrew's call: **A1.** Fresh-DB rebuild semantics are decisive. The audit trail is preserved by:

- PR #75's body remains on GitHub as a permanent record of the original intent.
- Git log shows `3acc328` (PR #75 squash) followed by this fix PR's commit; the diff is fully visible.
- This handoff and the attached postmortem document the failure, the rollback verification, and the fix.
